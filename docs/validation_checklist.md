## tv-backtest-api 検証チェックリスト（自動売買 bot パラメータ研究向け）

このチェックリストは、`tv-backtest-api` を自動売買 bot のパラメータ研究用として**どこまで信頼できるか**を検証するための項目を整理したものです。  
各項目について、検証日時・担当者・結果・メモを記録してください。

---

### 1. TradingView との差分確認

| --- | ---- | ------------- | ------ | ---- | ---------------------- | ---- |
| 1-1 | 価格系列の一致 | 同一シンボル・同一期間・同一足種でエクスポートした CSV の終値・高値・安値・出来高が TradingView のチャート表示と一致することを確認する。 |  |  |  |  |
| 1-2 | シグナルタイミングの一致 | シンプルな戦略（例: EMA クロス）を TradingView と本ツールの両方で実装し、**売買シグナル発生バーのタイミングと方向（ロング / ショート）が一致**するか確認する。 |  |  |  |  |
| 1-3 | 損益曲線の形状 | 上記シンプル戦略で、トレードの勝敗シーケンスと累積損益のカーブ形状が TradingView と同様の傾向になることを確認する（完全一致が難しい場合は差分理由をメモ）。 |  |  |  |  |

| 1-5 | タイムゾーンの取り扱い | TradingView のチャートタイムゾーンと、CSV / 本ツールが想定しているタイムゾーンが整合していることを確認する（ずれがある場合は変換ルールを明文化）。 |  |  |  |  |

### 4. Walk Forward の正当性確認

| No. | 観点 | 手順 / 期待値 | 実施日 | 担当 | 結果（OK / NG / 保留） | メモ |
| --- | ---- | ------------- | ------ | ---- | ---------------------- | ---- |
| 4-1 | ウィンドウ分割の妥当性 | `train_bars` / `test_bars` / `step_bars` の指定に対し、各 window の train / test 期間が期待どおりに分割されているか（重複・ギャップがないか）を確認する。 |  |  |  |  |
| 4-2 | 学習データと検証データの非重複 | 各 window において、**最適化に使った期間（train）と OOS 評価期間（test）が重なっていない**ことを確認する。 |  |  |  |  |
| 4-3 | 最適パラメータ適用の正しさ | 各 window の `best_params` を用いて独立に backtest を行い、`windows[oos_score]` と同じ OOS スコアになることを確認する。 |  |  |  |  |
| 4-4 | サマリ指標の算出 | `summary` の `avg_oos_score` / `median_oos_score` / `best_oos_score` / `worst_oos_score` / `success_windows` / `failed_windows` が `windows` の値から正しく集計されていることを手計算でスポットチェックする。 |  |  |  |  |
| 4-5 | 失敗ウィンドウの扱い | データ欠損や最適化失敗などで window が `failed` になった場合、その window がサマリの分母にどう影響するか（除外 or 0 扱いなど）が設計どおりであることを確認する。 |  |  |  |  |

### 5. 壊れた CSV / 壊れた strategy への耐性確認

| No. | 観点 | 手順 / 期待値 | 実施日 | 担当 | 結果（OK / NG / 保留） | メモ |
| --- | ---- | ------------- | ------ | ---- | ---------------------- | ---- |
| 5-1 | 欠損値を含む CSV | 価格や出来高に欠損値・NaN・空行を含む CSV を与えた場合に、**明示的なエラー**として検出されるか、もしくはスキップ・補完などのルールが一貫して適用されるか確認する。 |  |  |  |  |
| 5-2 | 異常なタイムスタンプ | タイムスタンプが逆順・重複・大きなギャップを持つ CSV を与えた場合の挙動（ソート・警告・エラー）が期待どおりか確認する。 |  |  |  |  |
| 5-3 | 不正カラム構成 | 必須カラム不足、余分なカラム、型不一致がある CSV で、**どのようなバリデーションエラーが出るか**を確認し、メッセージが原因特定に十分かどうか評価する。 |  |  |  |  |
| 5-4 | strategy 実装エラー | 例外を投げる strategy・0 で割る strategy など明らかなバグを含む strategy を実行し、API が 500 エラーやカスタムエラーとして適切にハンドリングし、システム全体が落ちないことを確認する。 |  |  |  |  |
| 5-5 | 無限ループ・極端な計算量 | 無限ループや非常に重い処理を含む strategy を実行した場合、タイムアウトやキャンセルなどで**制御が戻ってくるか**、あるいは監視・アラートの対象になるか確認する。 |  |  |  |  |
