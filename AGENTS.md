# AGENTS.md

このファイルは、このリポジトリで作業する AI エージェント向けの運用ルールです。  
主な対象は Cursor、ChatGPT 系エージェント、Codex 系エージェントです。

---

## 1. 基本方針

- 常に **最小スコープ** で対応すること
- ユーザー依頼や Issue に含まれない拡張は勝手に行わないこと
- 実装前に、まず関連ファイル・責務・影響範囲を確認すること
- 不明点があっても、既存コードから妥当な範囲で補完できるなら調査を優先すること
- 未確認事項を断定しないこと
- 大きい変更は 1 回で抱え込まず、必要なら分割を提案すること

---

## 2. 実装フロー

Issue ベースの作業では、原則として次の順で進めること。

1. Issue の内容を読む
2. 関連ファイルと影響範囲を調査する
3. 実装方針を簡潔に整理する
4. 最小スコープで実装する
5. テスト・確認を実行する
6. PR を作成する
7. 最終報告を返す

---

## 3. Issue / PR 運用ルール

- Issue ベースで作業する場合は、必ず対象 Issue 番号を特定してから着手すること
- ブランチ名には必ず Issue 番号を含めること
- 実装完了後は push だけで終わらせず、必ず PR を作成すること
- PR タイトルまたは本文に Issue 番号を明記すること
- PR 本文には必ず `Closes #[issue_number]` を含めること
- Issue 番号が不明な場合は、PR 作成前に確認すること
- Issue と無関係な追加修正は混ぜないこと
- 最終報告では、Issue 番号、ブランチ名、PR URL、実行したテストを必ず記載すること

### ブランチ名の推奨形式

- `feature/#123-hide-latest-optimization-result`
- `fix/#456-handle-partial-optimization-json`
- `chore/#789-update-pr-template`

厳密な形式はプロジェクト側の運用に従うこと。  
ただし、**Issue 番号を含めることは必須** とする。

### PR 本文テンプレート

```md
## Summary
- [変更内容を簡潔に記載]

## Issue
Closes #[issue_number]

## Changes
- [変更点1]
- [変更点2]

## Test
- [実行したテスト]

## Risks
- [未解決リスクがあれば記載]
```

---

## 4. コード変更時のルール

- 既存の命名規則、責務分担、ディレクトリ構成を尊重すること
- 無関係なリファクタや整形を混ぜないこと
- 既存 API / UI / DB 互換性を壊さないこと
- 変更理由が明確でない大規模リネームは避けること
- 1つの Issue で複数の関心事をまとめて変更しないこと

---

## 5. テスト・確認ルール

- 実装したら必ず何らかの確認を行うこと
- 可能であれば自動テストを実行すること
- テストが存在しない場合は、最低限の手動確認手順を明記すること
- テストが失敗した場合は、原因を確認し、修正後に再実行すること
- テスト不能な場合は、その理由を最終報告に明記すること

### 最終報告に含めること

- 変更ファイル一覧
- 実装内容の要約
- 実行したテスト
- テスト結果
- 未解決リスク
- PR URL

---

## 6. 調査・報告ルール

実装前の調査では、次を優先すること。

- 関連ファイル
- 呼び出し元 / 呼び出し先
- 型やデータ構造
- UI 表示箇所
- API 入出力
- 既存の類似処理

報告は実務的・簡潔・正確に行うこと。  
長い抽象説明は避け、次のような形を優先すること。

- 何を見たか
- どう判断したか
- 何を変更したか
- 何を確認したか
- 何が未解決か

---

## 7. 禁止事項

- Issue にない拡張を勝手に追加すること
- 影響範囲の大きい変更を無断で混ぜること
- テストせずに完了扱いにすること
- push のみで終わり、PR を作らないこと
- `Closes #[issue_number]` なしで PR を作ること
- 未確認の仕様を断定すること
- 無関係なファイルを大量に変更すること

---

## 8. Cursor での作業時の推奨動作

Cursor で作業する場合は、最初に以下を行うこと。

- Issue URL または Issue 番号を確認する
- 関連ファイルを探索する
- 実装前に簡潔な方針を出す
- 実装後にテストする
- PR を作成し、URL を返す

### Cursor に渡す指示文の推奨形式

```text
この issue を実装してください: [Issue URL]
必ず issue 番号を特定し、その issue に紐づくブランチで作業してください。
実装完了後は、必ず PR まで作成してください。
PR タイトルと本文には issue 番号を明記し、本文の先頭付近に `Closes #[issue_number]` を必ず入れてください。
push だけで終わらせず、PR 作成まで完了してください。
最初に関連ファイルを調査して、影響範囲と実装方針を簡潔にまとめてください。
その後、最小スコープで実装してください。
最後に、変更ファイル一覧、実行したテスト、PR URL、未解決リスクを報告してください。
```

---

## 9. このファイルの位置

この `AGENTS.md` は **リポジトリのルート** に配置すること。  
例:

```text
my-project/
├─ AGENTS.md
├─ README.md
├─ .github/
├─ src/
└─ ...
```

原則として、まずはルートに 1 つ置けばよい。  
特定ディレクトリ単位で追加ルールが必要になった場合のみ、別途ドキュメントを追加する。

---

## 10. 優先順位

ルールが競合する場合の優先順位は次の通り。

1. ユーザーの明示的な依頼
2. Issue の要件
3. この `AGENTS.md`
4. 既存コードベースの慣習

常に、ユーザー依頼と Issue のスコープを最優先すること。
